Gradle Build

DexGuard can be integrated in the Gradle build process of the Android SDK Tools (version 0.6.3, for Gradle version 1.8).

Setting up the Gradle build

You can enable DexGuard in the Android Gradle build by modifying the build.gradle file of your project: 
buildscript {
    repositories {
        flatDir { dirs '/usr/local/DexGuard5.4/lib' }
        mavenCentral()
    }
    dependencies {
        classpath ':dexguard:'
        classpath 'com.android.tools.build:gradle:0.6.3'
    }
}

apply plugin: 'dexguard'

android {
    .....
    buildTypes {
        debug {
            proguardFile plugin.getDefaultDexGuardFile('dexguard-debug.pro')
            proguardFile 'dexguard-project.txt'
            proguardFile 'proguard-project.txt'
        }
        release {
            proguardFile plugin.getDefaultDexGuardFile('dexguard-release.pro')
            proguardFile 'dexguard-project.txt'
            proguardFile 'proguard-project.txt'
        }
    }
}
Please make sure the repository path in the build script is set correctly for your system. Also make sure the :dexguard: dependency is listed first.

The standard Gradle build process will now use DexGuard's Gradle task. You'll see this in the console output: the line :dexguardRelease in the console replace all lines :dexRelease, :packageRelease, and :zipalignRelease.

You'll most comonly apply DexGuard to final application projects. This is the easiest and most effective approach, since it processes the combined code in a single step.

However, you can also apply DexGuard to library projects, if the library aar file or jar file is the final product: 

buildscript {
    repositories {
        flatDir { dirs '/usr/local/DexGuard5.4/lib' }
        mavenCentral()
    }
    dependencies {
        classpath ':dexguard:'
        classpath 'com.android.tools.build:gradle:0.6.3'
    }
}

apply plugin: 'dexguard-library'

android {
    .....
    debug {
        runProguard false
        proguardFile plugin.getDefaultDexGuardFile('dexguard-library-debug.pro')
        proguardFile 'dexguard-project.txt'
        proguardFile 'proguard-project.txt'
    }
    release {
        runProguard true
        proguardFile plugin.getDefaultDexGuardFile('dexguard-library-release.pro')
        proguardFile 'dexguard-project.txt'
        proguardFile 'proguard-project.txt'
    }
}
Note that Android library projects only have build types debug and release; the buildTypes container doesn't exist.

You should take some care with class encryption, as discussed for the class encryption option. Furthermore, tamper detection is only supported if the final application is also built and packaged using DexGuard.

You can find examples of working projects in the directory samples.

Building

The procedure for building Android applications remains the same. You can invoke gradle with the usual targets, such as installDebug and installRelease. Debug builds use debug settings, without optimization or obfuscation. Release builds use release settings, with full optimization and obfuscation. Both types of builds can optionally be signed. Both types of builds create application archives with aligned entries.

For compatibility with Ant and with Eclipse, you can configure your Gradle build to pick up custom configurations from the following files in your Android project: 

dexguard-project.txt         (for all build types)
dexguard-project-debug.txt   (for build type 'debug')
dexguard-project-release.txt (for build type 'release')
proguard-project.txt         (for backward compatibility with ProGuard)
The default version of the first file contains a number of settings that can be enabled to perform string encryption, class encryption, and to hide access to sensitive APIs.

Further tips

As before, you can specify some useful properties in the Gradle configuration file local.properties:
  • If you want to get verbose output for the entire build process, including DexGuard, you can run gradle with a more verbose log level: 
    gradle -info .....
  • If you want to sign your application, you can specify the necessary information in the build.gradle file of your project. For example, with hard-coded values: 
    android {
    .....
    signingConfigs {
        mySignature {
            storeFile     file('/home/user/.android/debug.keystore')
            storePassword 'android'
            keyAlias      'androiddebugkey'
            keyPassword   'android'
        }
    }

    defaultConfig {
        signingConfig signingConfigs.mySignature
    }
}

Reverting to the traditional Gradle build

You can go back to the basic Gradle build of the Android SDK by removing the changes from the build.gradle file of your project.
